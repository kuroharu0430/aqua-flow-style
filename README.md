## 1. Overview（概要）

このアプリケーションは、  
**Web上で要素を自由に配置・編集できる UI エンジンベースのレイアウト編集ツール**です。

グリッド上にパーツをドラッグして配置し、  
サイズ変更、複数選択、スタイル編集、Undo/Redo など  
直感的な操作でレイアウトを構築できます。

Webブラウザ上で動作し、  
Blazor / C# による高い操作性と柔軟な UI 制御を実現しています。

## 2. Features（機能一覧）

- **ドラッグ移動（Drag）**  
  要素をグリッド上で自由に移動できる。

- **リサイズ（Resize）**  
  右下のハンドルを掴んでサイズ変更が可能。

- **新規配置（Registering）**  
  テンプレートを選択してグリッドに新規配置。

- **複数選択（Selecting）**  
  範囲選択で複数の要素を一括操作。

- **コンテキストメニュー（右クリック）**  
  編集・削除などの操作を提供。

- **スタイル編集（Style Edit）**  
  ダイアログから色・枠線・フィールド値などを編集。

- **Undo / Redo（操作履歴）**  
  すべての操作が履歴として記録され、戻す・やり直しが可能。

- **オーバーラップモード（Push / Free）**  
  要素同士の衝突時の挙動を切り替え可能。

- **Trail Effect（移動軌跡）**  
  移動中のグリッドにエフェクトを表示。

## 3. Technical Background（技術背景）

このアプリケーションは、Blazor 上で高度な UI 操作を実現するために  
いくつかの技術的課題を解決しています。

### 1. Blazor の制約を補うための JS 連携（Drag/Resize の成立理由）
Blazor だけでは、ドラッグ操作に必要な以下の情報を取得できません：

- DOM の絶対座標（getBoundingClientRect）
- スクロール位置のリアルタイム監視（scrollTop / scrollLeft）
- マウスの移動方向（movementX / movementY）
- requestAnimationFrame による 60fps 更新

そのため本アプリでは、  
**不足する情報を JavaScript が補い、  
Blazor は状態管理と描画に専念する構造**を採用しています。

これにより、  
ネイティブアプリのような滑らかな Drag / Resize を実現しています。
  
### 2. Surface / Controller / State の役割分担（責務分離）

このアプリでは、UI が壊れず安定して動くように  
**役割を3つに分けています（責務分離）**。

- **Surface（見た目）**  
  マウス操作を受け取り、画面を描画する部分。  
  また、DOM 座標やスクロール量など **Blazor では取得できない情報を JS と連携して補う**役割も持ちます。  
  → 「見た目＋入力の受け口」を担当

- **Controller（操作の意味）**  
  今の操作がドラッグなのか、選択なのか、リサイズなのかを判断する部分。  
  Surface から受け取ったイベントを「意味のある操作」に変換します。  
  → 「操作のロジック」を担当

- **State（唯一の真実）**  
  レイアウトの位置・サイズ・選択状態など、UI のすべての状態を保持する部分。  
  Controller が操作した結果をここに反映し、Surface はこの状態を描画します。  
  → 「本当の状態（SSOT）」を担当

---

### この構造の一番のメリット
**メンテナンス性が圧倒的に高いこと。**

役割が分かれているため、

- Surface は「見た目と入力」だけ  
- Controller は「操作の意味」だけ  
- State は「状態」だけ  

を担当し、互いに干渉しません。

そのため、

- **新しい機能を追加しても他が壊れない（拡張性）**  
- **コードの追跡がしやすく、原因を特定しやすい（保守性）**  
- **Drag / Resize / Undo / Redo が自然に動く（安定性）**  
- **UI が常に正しい状態を保つ（信頼性）**

というメリットがあります。
  
### 3. Undo / Redo のための Snapshot 設計

このアプリでは、操作履歴を「差分（変更点だけ）」として記録する  
**Snapshot 方式**を採用しています。

これにより、以下のメリットがあります。

- **処理が軽い**  
  毎回全データを保存するのではなく、  
  「変わった部分だけ」を記録するため高速に動作します。

- **保存メモリが少ない**  
  レイアウト全体を丸ごと保存しないため、  
  Undo/Redo の履歴が増えてもメモリを圧迫しません。

- **どんな操作でも保存できる（汎用性）**  
  Drag / Resize / Style変更 / 選択状態など、  
  すべての操作を同じ仕組みで記録できます。

- **新しい操作を追加しても簡単（拡張性）**  
  「この操作の Snap はこれ」という  
  小さな定義を追加するだけで Undo/Redo に対応できます。

結果として、  
**正確で軽量な Undo / Redo を実現しつつ、  
機能追加にも強い構造**になっています。

  
### 4. 座標系の完全抽象化

ブラウザ上の座標は、  
**Page / Surface / Scroll / Relative / Absolute** など複数の基準があり非常に複雑です。

本アプリでは、これらの座標を **State がすべて統合して管理**することで、  
Controller が座標計算を一切意識せずに操作できるようにしています。

その結果、

- Drag / Resize / Selection が常に正しい位置で動く  
- スクロール中でも座標がズレない  
- Controller が「どこに動かすか」だけ考えればよくなる  
- 新しい機能を追加しても座標計算を触らなくて済む（保守性・拡張性）

というメリットがあります。
  
### Session 方式の最大のメリット
この方式の最大のメリットは、  
**Drag / Resize / Push / Swap が “数百行で成立するほど軽い” ことです。**

通常の UI エンジンでは、  
事前の Validate や例外処理が膨れ上がり、  
Drag 処理だけで数千行になることも珍しくありません。

本アプリでは、

- 操作前に状態を記録し  
- 実際に動かし  
- ダメなら即座に戻す  

というシンプルなロジックにより、  
複雑な操作を安全に扱いながら、  
**極めて軽量なコードで Drag を実現しています。**

これは従来の UI エンジンでは難しかった  
**「壊れないのに軽い」  
「複雑なのにシンプル」  
という構造を可能にしています。**
